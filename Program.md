# 🎯 Program.md (Hotel ECS Digital Twin Architect Program)

ไฟล์นี้ทำหน้าที่เป็นคู่มือสถาปัตยกรรมสูงสุด (Global Rules & Instructions) สำหรับควบคุมการทำงานและทักษะของ AI Agent (Antigravity/Jules) ภายในระบบจำลองสถานะห้องพัก (Digital Twin) และตู้ Phonik PBX

---

## 🚀 เป้าหมายสูงสุด (Ultimate Goal)
พัฒนาระบบตัวกลางส่งคำสั่งควบคุมการเปิด/ปิดไฟฟ้าในห้องพัก (PBX Connector) รวมถึงบันทึกชื่อผู้เข้าพัก ให้ทำงานได้อย่างราบรื่น แม่นยำ และปลอดภัย 100%

---

## 🏛️ สถาปัตยกรรมและทักษะ (Architecture & Skills)
สถาปัตยกรรมระบบได้รับการออกแบบมาในรูปแบบ **Modular Architecture (Class-based)** แบ่งการทำงานออกเป็น 3 เสาหลักที่เชื่อมต่อกัน:

1. **`PBXProtocolHandler` (ตัวแปลโปรโตคอล):**
   - แปลง Action สถานะไฟฟ้าหรือข้อมูลชื่อผู้เข้าพักระดับสูงเป็น ASCII command ตามที่ตู้สาขากำหนด (`..ROOMxxxx=y\r\n`)
2. **`ConnectionHandler` (ตัวจัดการเครือข่าย):**
   - ดูแลการรับส่งข้อมูลผ่าน Socket/Telnet
   - **Safety Constraints:** มีชั้น Safety Wrapper บล็อกคำสั่งที่เป็นภัยต่อตู้สาขาก่อนปล่อยออกสู่อุปกรณ์จริง
3. **`StateVerifier` (ตัวยืนยันสถานะ):**
   - ถอดรหัสการยืนยันสถานะที่ตอบกลับจากตู้สาขาเพื่อตรวจสอบความสอดคล้องกับค่าที่ส่งไป

---

## 🔄 โปรโตคอลการทำงาน (Harness Loop Protocol)
ในการสั่งงานฮาร์ดแวร์จริงหรือ Simulator ในแต่ละรอบ Agent จะต้องรันโปรเซสตามกรอบการทำงาน 4 ขั้นตอนอย่างเคร่งครัด:

```
┌──────────────┐      ┌──────────────┐      ┌──────────────┐      ┌──────────────┐
│    PLAN      │ ───> │     DO       │ ───> │   VERIFY     │ ───> │    DECIDE    │
│ วางขั้นตอน     │      │ สั่งงานผ่าน   │      │ ตรวจสอบสถานะ  │      │ ประเมิน/     │
│ เตรียมความพร้อม│      │ TCP/Socket   │      │ ไฟห้องพักจริง  │      │ ซ่อมแซมตัวเอง │
└──────────────┘      └──────────────┘      └──────────────┘      └──────────────┘
```

1. **PLAN (วางแผน):** ระบุคำสั่งและเป้าหมายที่ต้องการ พร้อมตรวจสอบความถูกต้องของพารามิเตอร์
2. **DO (ดำเนินการ):** ดำเนินการแปลงคำสั่งผ่าน `PBXProtocolHandler` และส่งคำสั่งผ่าน `ConnectionHandler`
3. **VERIFY (ยืนยัน):** วิเคราะห์ผลตอบกลับผ่าน `StateVerifier` เสมอ เพื่อยืนยันว่ากระบวนการสำเร็จจริง
4. **DECIDE (ตัดสินใจ):** 
   - หาก Verifier ยืนยันว่าสำเร็จ ให้บันทึกการส่งงานและปิดรอบ
   - หากล้มเหลว (เช่น ตรวจพบ Timeout หรือ Network loss) ให้ทำการเชื่อมต่อใหม่และรัน Loop ซ้ำตามนโยบาย (Retry Policy)
   - หากระบบตรวจพบข้อผิดพลาดรุนแรง (เช่น `==NACK` จากฮาร์ดแวร์) ให้ยกเลิกการ Retry (Abort) เพื่อรักษาความปลอดภัยของระบบโดยรวม
