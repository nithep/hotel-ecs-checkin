# คู่มือแนวทางการเข้าถึงอุปกรณ์และระบบเครือข่าย (Network Access & Architecture Guideline)

**ระบบ Smart Hotel Self Check-in/Check-out (Hotel ECS)**

คู่มือฉบับนี้อธิบายสถาปัตยกรรมการเข้าถึงอุปกรณ์เน็ตเวิร์คและระบบเซิร์ฟเวอร์ทั้งหมดของโปรเจกต์ โดยแบ่งแยกตามบทบาทของผู้ใช้งาน (Roles) และอธิบายกลไกการเชื่อมโยงข้อมูลจากภายนอกสู่ฮาร์ดแวร์ภายใน

---

## 1. ภาพรวมสถาปัตยกรรมเครือข่าย (Network Architecture Overview)

ระบบ Hotel ECS ใช้สถาปัตยกรรมเครือข่ายแบบผสมผสาน (Hybrid Network) เพื่อให้เกิดความสมดุลระหว่าง **ความสะดวกในการใช้งาน (Accessibility)** และ **ความปลอดภัยสูงสุด (Security)** ประกอบด้วย 2 ท่อหลักที่ทำงานขนานกัน:

1. **ท่อสาธารณะที่ปลอดภัย (Cloudflare Tunnel):** สำหรับให้บริการ Web Application แก่ผู้ใช้งานทั่วไป โดยซ่อน IP จริงของเซิร์ฟเวอร์
2. **ท่อส่วนตัวหลังบ้าน (WireGuard VPN):** สำหรับผู้ดูแลระบบ (Admin) เพื่อเข้าควบคุมระบบปฏิบัติการ (OS) และฮาร์ดแวร์ตู้สาขาโดยตรง

---

## 2. ช่องทางการเข้าถึงแบ่งตามกลุ่มผู้ใช้งาน (Access Channels by User Roles)

### 👤 2.1 แขกผู้เข้าพัก (Hotel Guests)
- **อุปกรณ์ที่รองรับ:** สมาร์ทโฟน, แท็บเล็ต (ผ่าน Mobile Browser เช่น Safari, Chrome)
- **ช่องทางการเชื่อมต่อ:** `Public Internet` ➡️ `Cloudflare Tunnel` ➡️ `Frontend Web (Self Check-in)`
- **รูปแบบการใช้งาน:** 
  - แขกสแกน QR Code หน้าห้องพัก หรือเข้าผ่าน URL ที่กำหนด
  - ข้อมูลจะถูกเข้ารหัสผ่าน Cloudflare วิ่งเข้าสู่บอร์ด Raspberry Pi 4 อย่างปลอดภัย
- **สิทธิ์การเข้าถึง (Permissions):** สิทธิ์ระดับต่ำสุด (Guest) สามารถโต้ตอบได้เฉพาะหน้า UI ระบบ Check-in/Check-out เท่านั้น **ไม่มีสิทธิ์** มองเห็น IP ภายใน หรืออุปกรณ์เน็ตเวิร์คใดๆ

### 👨‍💼 2.2 พนักงานต้อนรับ / ผู้จัดการ (Staff / Managers)
- **อุปกรณ์ที่รองรับ:** คอมพิวเตอร์หน้าฟร้อนท์, แท็บเล็ตของโรงแรม
- **ช่องทางการเชื่อมต่อ:** `Public Internet` ➡️ `Cloudflare Access (Zero Trust)` ➡️ `Admin Dashboard`
- **รูปแบบการใช้งาน:**
  - พนักงานเปิด Web Dashboard ผ่านเบราว์เซอร์
  - ระบบจะบังคับให้ยืนยันตัวตน (Authentication) ผ่านนโยบายของ Cloudflare Access (เช่น Login ด้วย Email ของโรงแรม หรือ OTP) ก่อนถึงจะเห็นหน้าจอจัดการห้องพัก
- **สิทธิ์การเข้าถึง (Permissions):** สิทธิ์ระดับกลาง สามารถสั่งเปิด/ปิดไฟในห้อง (ผ่าน UI), ดูสถานะห้องพัก, แต่ **ไม่สามารถ** เข้าไปแก้ไขโค้ดหรือเข้าหน้าต่าง Command Line ของเซิร์ฟเวอร์ได้

### 🛠️ 2.3 ผู้ดูแลระบบ / ช่างเทคนิค (System Administrators / Technicians)
กลุ่มนี้มีความจำเป็นต้องเจาะลึกทะลุระบบป้องกันเพื่อเข้าไปซ่อมบำรุงหรือตั้งค่าระดับลึก (Deep Level)
- **อุปกรณ์ที่รองรับ:** คอมพิวเตอร์ PC ของ Admin (Windows), สมาร์ทโฟน/แท็บเล็ต ของช่าง
- **ช่องทางการเชื่อมต่อ (มี 2 รูปแบบ):**
  1. **เข้าจัดการ Web/Cloud:** ผ่านอินเทอร์เน็ตปกติ ➡️ `Cloudflare Dashboard` (เพื่อจัดการ Policy, DNS)
  2. **เข้าจัดการ Hardware/Server (ทางตรง):** `Admin Device` ➡️ `WireGuard VPN Tunnel` ➡️ `Raspberry Pi 4 (Local IP: 10.x.x.x / 192.x.x.x)` ➡️ `ตู้สาขา Phonik PBX`
- **รูปแบบการใช้งาน (บนคอมพิวเตอร์ / สมาร์ทโฟน):**
  - ช่างต้องเปิดแอป **WireGuard** และกด Connect เพื่อเชื่อมท่อ VPN 
  - เมื่อเชื่อมต่อแล้ว จะเสมือนว่าช่างนั่งนำสาย LAN ไปเสียบตรงๆ เข้ากับวงเน็ตเวิร์คของโรงแรม
  - สามารถใช้โปรแกรม SSH (เช่น Termius, PuTTY) มุดเข้าบอร์ด Pi 4 หรือส่งคำสั่ง Serial ลงตู้สาขา PBX ได้โดยตรง
- **สิทธิ์การเข้าถึง (Permissions):** สิทธิ์ระดับสูงสุด (Root/Superadmin) ควบคุมได้ทั้ง Software และ Hardware

## 3. กลไกการเชื่อมโยงและบริหารจัดการทับซ้อน (Integration & Split Tunneling)

สำหรับเครื่องคอมพิวเตอร์ของผู้ดูแลระบบ (Admin) มักจะมีการใช้งาน **Cloudflare WARP (Zero Trust Client)** เพื่อป้องกันภัยคุกคามทางไซเบอร์ ซึ่งจะทำงานทับซ้อนกับ **WireGuard VPN** 

เพื่อให้ระบบทำงานร่วมกันได้โดยอินเทอร์เน็ตไม่ตัด (ปัญหา NCSI สับสน) ระบบถูกออกแบบกลไกดังนี้:

1. **Split Tunneling Rule:** บน Cloudflare Dashboard ได้มีการตั้งค่า Exclude (ยกเว้น) ไอพีวงภายในของโรงแรม (เช่น `10.0.0.0/8`, `192.168.0.0/16`) ออกจากท่อของ WARP 
2. **การวิ่งของ Data:**
   - เมื่อ Admin เปิดเว็บดู YouTube หรือเข้าเว็บทั่วไป ข้อมูลจะถูกบังคับวิ่งเข้าท่อรักษาความปลอดภัยของ **Cloudflare WARP**
   - เมื่อ Admin พิมพ์ IP ของเซิร์ฟเวอร์ Pi 4 เพื่อรีโมทเข้าไปแก้บั๊ก (เช่น 192.168.1.10) ข้อมูลจะแยกทางวิ่งลงท่อ **WireGuard** เพื่อพุ่งตรงเข้าเครือข่ายโรงแรมทันที
3. **การตั้งค่า Windows NCSI:** เครื่อง Admin ต้องรันคำสั่ง PowerShell บังคับให้ Windows ไม่ต้องเช็คสถานะเน็ตเวิร์คซ้ำซ้อนผ่าน DNS ที่ตีกันระหว่าง 2 VPN เพื่อป้องกันอาการ Wi-Fi Disconnect

---

> **หมายเหตุ:** โครงสร้างนี้อ้างอิงจากหลักการ "Zero Trust" (ไม่ไว้ใจใครเลยแม้จะอยู่ในเครือข่ายเดียวกัน) และ "Defense in Depth" (การป้องกันหลายชั้น) ทำให้ระบบมีความคงทนต่อการถูกแฮ็กจากภายนอก แต่ยังให้ความคล่องตัวกับทีมช่างในการรีโมทเข้าซ่อมบำรุงจากระยะไกลได้อย่างรวดเร็ว
