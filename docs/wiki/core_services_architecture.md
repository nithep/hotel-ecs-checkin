# 🧠 สถาปัตยกรรมบริการหลักและการแก้ไขปัญหาเชิงลึก (สำหรับช่างเทคนิค)

เอกสารฉบับนี้จัดทำขึ้นเพื่อเป็น **"แผนที่โครงสร้างระดับลึก (Deep Dive)"** ให้ช่างเทคนิคและผู้ดูแลระบบเข้าใจหัวใจของระบบที่ต้องรัน 24/7 เพื่อให้สามารถวิเคราะห์ปัญหาหน้างานได้ตรงจุด รวดเร็ว โดยไม่ต้องพึ่งพา AI ในการสุ่มหาปัญหาแต่แรก (ช่วยประหยัดเวลาและทรัพยากร)

---

## 🛑 1. การวิเคราะห์ปัญหาสถานะเครือข่าย VPN (กรณีศึกษา `hotel-admin`)

บ่อยครั้งที่ช่างเทคนิคอาจสับสนกับข้อความแจ้งเตือนของ Windows ทึ่ระบุว่า:
> **"hotel-admin: ไม่มีการเข้าถึงอินเทอร์เน็ต"**

### จะมีผลอย่างไร? เมื่อไร?
1. **ในสภาวะปกติ (ไม่มีผลกระทบต่อโรงแรม):** 
   - ข้อความนี้เป็น **เรื่องปกติ** เพราะอุโมงค์ `hotel-admin` (WireGuard) เป็นแบบ Split-Tunnel (อนุญาตให้วิ่งเฉพาะเครือข่ายภายใน เช่น 10.0.0.0/24) Windows จึงมองว่ามันออกเน็ตไม่ได้
   - **สิ่งที่ต้องทำ:** ไม่ต้องทำอะไร ปล่อยไว้ แขกยังสามารถเข้าเว็บเช็คอินได้ปกติ และ PBX ก็ทำงานปกติ
2. **ในสภาวะผิดปกติ (มีผลเฉพาะผู้ดูแลระบบ):**
   - หากทดสอบ `ping 10.0.0.1` แล้วขึ้น `Request Timed Out` แปลว่า VPN ดับไปจริงๆ
   - **ผลกระทบ:** แอดมิน (Matebook) จะไม่สามารถ **รีโมท (SSH) หรือเข้าไปดู Log เซิร์ฟเวอร์จากระยะไกลได้** 
   - **แขกได้รับผลกระทบไหม?:** **ไม่ได้รับผลกระทบ!** เพราะแขกเข้าผ่านเส้นทาง Cloudflare Tunnel (`hotel.nithep.com`) ซึ่งทำงานแยกกันเด็ดขาด

**📌 กฎเหล็กสำหรับช่าง:** หากระบบแขก (Check-in) มีปัญหา อย่าเพิ่งไปงมแก้ VPN ของแอดมิน ให้พุ่งเป้าไปเช็คที่ **Cloudflare Tunnel** และ **API Server** บน Pi 4 ก่อนเสมอ!

## ⚙️ 2. กลไกการทำงานของ 4 ทหารเสือ (24/7 Core Services)

การทำความเข้าใจ 4 ระบบนี้ จะช่วยให้ช่างสามารถนำระบบไป **"ต่อยอด (Scale & Extend)"** ในอนาคตได้อย่างไร้ขีดจำกัด:

### 📡 2.1 API (Express.js) - "ระบบประสาทส่วนกลาง"
- **การทำงาน:** เป็นพนักงานต้อนรับที่คอยรับคำสั่ง HTTP/REST ตลอด 24 ชม. ไม่ว่าแขกจะสแกน QR Code หรือแอดมินกดผ่าน AppSheet คำสั่งทั้งหมดต้องวิ่งมาเข้าที่นี่
- **หากขัดข้อง:** หน้าเว็บจะโหลดหมุนค้าง หรือแจ้งเตือน 502 Bad Gateway
- **ไอเดียต่อยอด (Extension):** ในอนาคตเราสามารถเปิด API Endpoint ใหม่ให้นักพัฒนาภายนอก (Open API) เช่น ให้แอปจองห้องพัก Agoda/Booking ยิงคำสั่งจองเข้ามาระบบเราได้โดยตรง หรือเชื่อมกับ **Smart Door Lock** เพื่อออกรหัสผ่านห้องพัก

### 💾 2.2 SQLite (`hotel.db`) - "ความทรงจำถาวร"
- **การทำงาน:** ฐานข้อมูลขนาดเบาที่เก็บสถานะว่าห้องไหนว่าง หรือมีคนอยู่
- **ความสำคัญ 24/7:** หากไฟดับหรือบอร์ด Pi 4 รีสตาร์ทตัวเอง ทันทีที่ระบบตื่นขึ้นมา มันจะวิ่งไปอ่าน SQLite เพื่อจำได้ว่าต้องสั่งเปิดไฟห้องไหนค้างไว้บ้าง 
- **ไอเดียต่อยอด (Extension):** หากโรงแรมขยายตัวใหญ่ขึ้น มีสาขามากขึ้น สามารถย้าย (Migrate) จาก SQLite ไปใช้ Cloud Database (เช่น PostgreSQL/Supabase) เพื่อรวมศูนย์ข้อมูลหลายสาขาไว้บนแดชบอร์ดเดียวได้ง่ายๆ

### ⏰ 2.3 Node-cron - "นาฬิกาชีวิตและแม่บ้าน"
- **การทำงาน:** ระบบตั้งเวลาอัตโนมัติภายใน Backend (คล้ายนาฬิกาปลุก) ปัจจุบันใช้สำหรับสรุปยอดส่งรายงานตอน 23:55 น.
- **หากขัดข้อง:** ระบบหลักไม่พัง แต่รายงานตอนดึกจะไม่ส่ง
- **ไอเดียต่อยอด (Extension):** สามารถเขียนโค้ดเพิ่มให้ Cronjob คอยเช็คเวลาตอน **12:00 น. (เที่ยงตรง)** ของทุกวัน หากพบว่าห้องไหนหมดเวลาพักและแขกไม่ยอมกด Check-out ระบบสามารถ **สั่งตัดไฟห้องนั้นอัตโนมัติ (Auto-Eviction)** ทันที!

### 📣 2.4 Webhook (Google Workspace) - "กระบอกเสียง"
- **การทำงาน:** เส้นทางส่งข้อความออกแบบทิศทางเดียว (Push Notification) ออกไปยัง Google Chat และ Google Sheets
- **หากขัดข้อง:** แขกยังเช็คอินและไฟเปิดได้ปกติ แต่พนักงานฟรอนต์จะไม่ได้รับไลน์/แชทแจ้งเตือน
- **ไอเดียต่อยอด (Extension):** รูปแบบ Webhook นี้เป็นมาตรฐานสากล หากวันหน้าโรงแรมเลิกใช้ Google Chat ก็แค่เปลี่ยน URL Webhook ไปยิงเข้า **LINE Notify, Slack, หรือระบบส่ง SMS (Twilio)** ได้ทันทีโดยไม่ต้องรื้อสถาปัตยกรรมหลักเลย
