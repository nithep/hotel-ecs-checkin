# 🌐 คู่มือการตรวจสอบความสมบูรณ์และการติดตั้งสถานที่ใหม่ (New Site Commissioning Guide)

เอกสารฉบับนี้จัดทำขึ้นเพื่อใช้เป็นแนวปฏิบัติมาตรฐาน (Standard Operating Procedure - SOP) สำหรับช่างเทคนิคและทีมพัฒนาในการย้ายอุปกรณ์หรือติดตั้งระบบ **Smart Hotel Self Check-in Gateway** ณ สถานที่บริการแห่งใหม่ เพื่อความรวดเร็ว แม่นยำ และปลอดภัยสูงสุดในรูปแบบ Plug & Play

## 📋 1. ขั้นตอนการบูตระบบเครือข่ายหน้างาน (Network Provisioning Checklist)

เมื่อถึงสถานที่ใหม่ การจัดลำดับการต่อเชื่อมเครือข่ายให้ทำงานแบบไร้หน้าจอ (Headless) มีความสำคัญสูงสุด:

### 🔌 ก. ระบบสายแลนหลัก (LAN DHCP Bootstrapping)
1. **เชื่อมต่อสายแลน**: เสียบสาย LAN จากตัวบอร์ด Raspberry Pi 4 เข้ากับสวิตช์วงเครือข่ายหลักของโรงแรม
2. **รับไอพีอัตโนมัติ**: บอร์ดจะทำการเปิดอินเทอร์เฟซ `eth0` เพื่อขอ IP Address ผ่าน DHCP Server ของเร้าเตอร์โรงแรม
3. **ค้นหาไอพีเพื่อรีโมท**: ช่างสามารถค้นหาไอพีเพื่อเข้าผ่าน SSH ด้วยคำสั่ง:
   ```bash
   ssh admin@pi-4.local
   # หรือใช้ Advanced IP Scanner ในวงเพื่อหาอุปกรณ์ "Raspberry Pi Foundation"
   ```
4. **ระบบ IP Reservation (สำคัญที่สุด)**: ติดต่อฝ่ายไอที of โรงแรมเพื่อทำ **DHCP Reservation (Static DHCP)** บนเร้าเตอร์หลัก โดยผูก MAC Address ของแลนบอร์ดเข้ากับ IP คงที่ (เช่น `192.168.1.109`) เพื่อป้องกันไม่ให้ไอพีเปลี่ยนภายหลัง ซึ่งจะส่งผลให้หน้าสแกน QR Code ใช้งานไม่ได้

### 📶 ข. ระบบไวไฟสำรองและการตั้งค่า (Wi-Fi Failover Configuration)
ในกรณีที่จำเป็นต้องใช้ไวไฟสำรอง หรือตั้งค่าไวไฟโรงแรมสำหรับการสลับการใช้งานอัตโนมัติ (Failover):
1. **รีโมทเข้าไปที่บอร์ดผ่าน LAN** สำเร็จแล้ว สั่งเปิดระบบเครื่องมือจัดการเครือข่าย:
   ```bash
   sudo nmtui
   ```
2. เลือก **"Activate a connection"** -> สแกนหาไวไฟของโรงแรม -> ใส่รหัสผ่าน WPA/WPA2 Key -> ตกลง
3. **ตรวจสอบตาราง Routing และ Metric**:
   ```bash
   ip route
   ```
   * *มาตรฐานลำดับความสำคัญ (Priority Metric)*:
     * สาย LAN (`eth0`) ต้องมี Metric ต่ำกว่า (เช่น `100`) -> เป็นเส้นทางหลัก
     * ไวไฟ (`wlan0`) ต้องมี Metric สูงกว่า (เช่น `600`) -> เป็นเส้นทางสำรอง
     * เมื่อสายแลนหลุด ข้อมูลจะสลับไปวิ่งผ่านไวไฟอัตโนมัติโดยไม่มี downtime ของระบบ

## 🐙 2. การตรวจสอบความสมบูรณ์ของ Git (Git Integrity Verification)

ระบบมีการใช้ Git ในการพัฒนาและซิงค์ข้อมูลบน Pi 4 โดยมีการแยกโครงสร้างโฟลเดอร์ไว้เพื่อความปลอดภัย:

### 🔍 ก. โครงสร้างโฟลเดอร์บน Pi 4
* **`/home/admin/hotel-ecs-checkin`**: ไดเรกทอรี Git หลัก (เชื่อมกับ Github) มีสิทธิ์ในการดึงชุดคำสั่งและอัปเดตเวอร์ชัน
* **`/home/admin/Hotel-ECS`**: ไดเรกทอรีรันระบบจริง (Production & Runtime) ซึ่งไม่มีไฟล์ `.git` ซ่อนอยู่ เพื่อป้องกันสิทธิ์การเขียนทับโดยไม่ตั้งใจและการโจมตีข้ามไดเรกทอรี

### 🔄 ข. ขั้นตอนการตรวจสอบและซิงค์โค้ดเวอร์ชันล่าสุด
เมื่อติดตั้งในสถานที่ใหม่ ทีมพัฒนาต้องตรวจสอบว่าโค้ดมีความสดใหม่และไม่มีการขัดแย้ง:
1. **ตรวจสอบความสมบูรณ์ของ Git**:
   ```bash
   cd /home/admin/hotel-ecs-checkin
   git status
   git remote -v
   ```
2. **ดึงอัปเดตล่าสุดจาก Remote Repository**:
   ```bash
   git pull origin master
   ```
3. **การนำโค้ดขึ้นระบบ Production (Deployment)**:
   หากมีการปรับปรุงโค้ดในไดเรกทอรี Git ให้ทำการทดสอบความถูกต้องในสภาพแวดล้อมทดสอบก่อน จากนั้นจึงคัดลอกไฟล์ขึ้นมาทดแทนใน `/home/admin/Hotel-ECS` และสั่งรีสตาร์ทบริการผ่าน PM2:
   ```bash
   pm2 restart hotel-backend
   pm2 restart hotel-pbx-connector
   ```

## 📞 3. การตรวจสอบระบบเชื่อมต่อตู้สาขา (Phonik PBX Connection Check)

ระบบเชื่อมต่อกับตู้สาขา Phonik ผ่านเครือข่าย TCP/IP (หรือ Telnet Protocol) การยืนยันความพร้อมของตู้สาขาสถานที่ใหม่ทำได้ดังนี้:

### 🔍 ก. ตรวจสอบการเปิดพอร์ตของตู้ PBX จริง
รันคำสั่ง Netcat บน Pi 4 เพื่อตรวจสอบว่าตู้สาขาของจริงออนไลน์และเปิดพอร์ต Telnet หรือไม่:
```bash
# ตรวจสอบ IP ตู้สาขาจริง (มาตรฐานคือ 192.168.1.91 พอร์ต 23)
nc -z -w 3 192.168.1.91 23
echo $?
```
* **หากตอบกลับสำเร็จ (สถานะ 0 หรือ True)**: เครือข่ายเชื่อมไปยังตู้สาขาได้ปกติ
* **หาก Timeout / Connection Refused**: ตรวจสอบการเข้าหัวสาย LAN การตั้งค่าไอพีของตู้ PBX หรือสายต่อเชื่อมแปลงพอร์ต

### 📝 ข. การตรวจสอบ Environment Config (`.env`)
ตรวจสอบคอนฟิกหลังบ้านว่าผูกอยู่กับตู้สาขาจริงอย่างถูกต้องในไฟล์ `/home/admin/Hotel-ECS/backend/.env`:
```env
PORT=3000
PBX_MODE=tcp
PBX_HOST=192.168.1.91
PBX_PORT=23
```
* *หมายเหตุ*: หากตู้สาขาในสถานที่ใหม่ใช้ไอพีหรือพอร์ตอื่น ให้แก้ไขไฟล์ `.env` นี้ให้ตรงกับหน้างานจริง จากนั้นรีสตาร์ทแอปหลังบ้านเพื่อโหลดการตั้งค่าใหม่

## 🔒 4. ขั้นตอนการทดสอบการใช้งานแบบปิดลูป (Closed-Loop Validation)

ก่อนการเปิดใช้จริงให้กับทางโรงแรม (Go-Live) ทีมพัฒนา**ต้อง**ทำการรันทดสอบระบบเสมือนและระบบจริงผ่านโมดูลความปลอดภัย:

### 🧪 ก. การรันทดสอบ Harness Loop (ในโฟลเดอร์ worker)
บนตัวบอร์ด Pi 4 ให้รันสคริปต์ตรวจสอบลูปข้อมูลแบบ Plan-Do-Verify-Decide เพื่อยืนยันว่าการแลกเปลี่ยนโปรโตคอลและ Safety Wrapper ทำงานสมบูรณ์:
```bash
cd /home/admin/Hotel-ECS/worker
python3 harness_loop.py --tcp
```
* **เกณฑ์การประเมิน**:
  * ระบบต้องจำลองส่งคำสั่งและอ่านสถานะตอบกลับได้สำเร็จ 100%
  * ระบบความปลอดภัย (Safety Wrapper) ต้องสามารถสกัดกั้นและขึ้นสถานะ `[SAFETY BLOCK]` เมื่อมีการป้อนคำสั่งอันตรายสำเร็จ

### 💓 ข. การอ่านบันทึก Heartbeat
ตรวจสอบล็อกของ backend เพื่อดูสถานะการเชื่อมต่อกับ PBX อย่างต่อเนื่อง:
```bash
pm2 logs hotel-backend --lines 10
```
* ต้องพบข้อความ `[PBX] 💓 Heartbeat OK` แสดงอย่างสม่ำเสมอ

## 🛠️ 5. เช็คลิสต์ตรวจสอบบริการเสริมและการเข้าถึงระยะไกล (Cloud Tunnel)

* **บริการเชื่อมต่อภายนอก (Cloud / SSH Tunnel)**: ตรวจสอบสถานะการเชื่อมต่อสำหรับช่วยให้โปรแกรมเมอร์สามารถรีโมทเข้ามาวิเคราะห์ระบบจากส่วนกลางได้:
  ```bash
  pm2 show hotel-tunnel
  ```
* **การเชื่อมต่ออินเทอร์เน็ตภายนอก (LINE API / Webhook)**:
  ```bash
  curl -I -s --connect-timeout 3 https://api.line.me | head -n 1
  ```
  * ต้องได้สถานะตอบกลับ `HTTP/2 301` หรือ `200` ยืนยันว่าสามารถคุยกับเซิร์ฟเวอร์ LINE ได้สำเร็จ
