# ทักษะ State Verifier (Loop Gatekeeper)

ทักษะนี้มีบทบาทเสมือนเป็น "ยามเฝ้าประตู" ของกระบวนการสั่งงานฮาร์ดแวร์ ทำหน้าที่ยืนยันว่าคำสั่งที่ส่งไปตู้ Phonik PBX นั้นทำงานสำเร็จและตรงตามความคาดหวังจริงหรือไม่ โดยอ้างอิงจากข้อมูล Log หรือ Response ที่ตอบกลับมา

## 🔍 กฎการตรวจสอบ (Verification Logic)

1. **การยืนยันผลสำเร็จ (ACK / Command Accepted)**:
   - ตู้สาขาจะตอบกลับคำสั่งที่ประมวลผลสำเร็จด้วย `==` ตามด้วยรายละเอียดของคำสั่งและค่าที่ตั้ง เช่น `==ROOM0101=1` หรือ `==NAME0101=John Doe`
   - หากคำตอบมีข้อมูลตรงกับคำสั่งที่ส่งไป ให้ถือว่าการตั้งค่าสำเร็จ (Success)

2. **การแจ้งปฏิเสธคำสั่ง (NACK / Command Rejected)**:
   - ตู้สาขาตอบกลับด้วยคำสั่งผิดพลาด เช่น `==NACK`
   - เมื่อตรวจพบ `==NACK` ระบบต้องระบุสถานะไม่สำเร็จ (Failed) พร้อมวิเคราะห์หาสาเหตุ (เช่น ห้องพักไม่มีอยู่จริง, คำสั่งพิมพ์ผิด หรือฮาร์ดแวร์ที่ฝั่งห้องพักไม่ตอบสนอง)

3. **กรณีไม่มีการตอบสนอง (Timeout / Silent Fail)**:
   - หากส่งคำสั่งไปแล้วไม่มีการตอบกลับภายในระยะเวลาที่กำหนด (เช่น 1-3 วินาที) ให้ถือเป็น Timeout (Failed)
   - อาจเกิดจากปัญหาเครือข่ายหลุด หรือตู้สาขาปิดอยู่

## 🔄 กลไกจัดการตนเองเมื่อเกิดข้อผิดพลาด (Self-Healing Loop)

เมื่อ Agent ประสบกับผลลัพธ์ที่เป็นล้มเหลว (NACK หรือ Timeout) ให้ปฏิบัติตามแนวทาง Self-Healing ดังนี้:

```
[ส่งคำสั่ง] ──> [ตรวจสอบด้วย Verifier]
                    │
           ┌────────┴────────┐
        (สำเร็จ)           (ล้มเหลว/Timeout)
           │                 │
     [จบกระบวนการ]    [ตรวจสอบจำนวนครั้งที่ Retry]
                             │
                    ┌────────┴────────┐
             (ยังไม่เกินกำหนด)    (เกินกำหนด / Max Retries)
                    │                 │
             [หน่วงเวลาสั้นๆ]     [แจ้งเตือนระบบล้มเหลว]
             [ทำการ Retry]        [บันทึก Log ลง /wiki]
```

1. **Retry Policy (การส่งซ้ำ)**:
   - หากเกิด Timeout ให้หน่วงเวลาเล็กน้อย (Exponential Backoff เช่น 500ms, 1000ms) แล้วทำการส่งคำสั่งเดิมซ้ำ
   - กำหนดให้ทำซ้ำสูงสุดไม่เกิน **3 ครั้ง**
2. **การวินิจฉัยปัญหาแบบแยกแยะ (Diagnostics)**:
   - หากได้ NACK ทุกครั้งจากการสั่งงานห้องเดิม (เช่น ห้อง 103 ตอบ NACK ตลอดในระหว่างทดสอบ) ให้บันทึกว่าเป็น Hardware Fault เฉพาะห้องนั้น และหลีกเลี่ยงการสแปมตู้สาขา
   - หากเป็น Timeout ทุกห้อง ให้สันนิษฐานว่าการเชื่อมต่อเครือข่าย (Socket) มีปัญหา ให้ลอง Reconnect ไปที่ตู้สาขาก่อนทำรายการต่อ
3. **การบันทึกสรุป (Documentation)**:
   - เมื่อเกิดเหตุการณ์ล้มเหลวที่แก้ไขไม่ได้ ให้ Agent เขียนรายงานรายละเอียดลงในโฟลเดอร์ `/wiki` หรือ `/docs/log.md` เสมอ เพื่อเก็บเป็นประวัติการทดสอบระบบ
